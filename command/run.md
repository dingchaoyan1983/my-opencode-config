description: 执行 /plan 命令生成的 plan 文件中定义的各个 Task
---

执行 `/plan` 命令生成的任务计划文档（plan 文件）中定义的各个 Task，按顺序完成代码改动并验证。

**输入**：`$ARGUMENTS` 是 plan 文件的路径或文件名。如果为空，优先使用**当前对话中最近一次生成或引用的 plan 文件**（如本会话中最近执行 `/plan` 生成的文件、或对话中最近提到路径的 plan 文件）；若当前对话中未出现过任何 plan 文件，再用 **read 工具**读取当前项目 `.opencode/plan/` 目录，用 **question 工具**列出其中所有 `.md` 文件供用户选择；如果目录不存在或为空，提示用户先使用 `/plan` 命令生成计划，然后停止。

在确定要执行的 plan 文件前不要继续。

---

## 执行流程

### 1. 定位 plan 文件

按以下顺序解析 `$ARGUMENTS`：

- 绝对路径 → 直接使用
- 相对路径 → 相对于当前项目根目录解析
- 仅文件名（如 `2026-06-21-admin-start-prod.md` 或 `2026-06-21-admin-start-prod`）→ 在 `.opencode/plan/` 目录下查找，自动补全 `.md` 后缀
- 为空 → 优先使用**当前对话中最近一次生成或引用的 plan 文件**（如本会话中最近 `/plan` 生成的文件、或对话中最近提到路径的 plan 文件）；若当前对话中未出现过任何 plan 文件，再用 **read 工具**读取 `.opencode/plan/` 目录，用 **question 工具**列出所有 plan 文件让用户选择

定位失败时停止并提示用户。

### 2. 读取并解析 plan

使用 **read 工具**读取 plan 文件完整内容，识别：

- `# 标题`：任务总体目标（执行参考，不做改动）
- `## 现状`：背景说明（执行参考，不做改动）
- `## Task N: <描述>`：需要执行的任务列表，按 N 升序

解析每个 Task 章节中的：

- 文字说明：任务要做什么、参照哪个文件
- 代码块（带语言标记如 ```js / ```json / ```tsx）：需要创建或写入的文件内容
- 参数来源列表（`- \`参数名\` -- 来源`）：执行时需回查来源文件核对的关键配置

### 3. 逐个执行 Task

对每个 Task，按顺序执行：

1. **宣告**：输出当前正在执行的 Task 编号与描述
2. **并行探索参照文件**：如 Task 文字说明中提到"参照 xxx 文件"，先用 **read 工具**并行读取这些参照文件，理解现有模式与约定
3. **执行改动**：
   - 创建新文件 → 使用 **write 工具**，内容直接采用 plan 代码块中的内容
   - 修改现有文件 → 先用 **read 工具**读取目标文件，再用 **edit 工具**做精确字符串替换
   - 多个独立的文件创建/修改可在同一条消息中并行发起工具调用
4. **核对参数来源**：对照 plan 中的"参数来源"列表，回查每个来源文件确认值一致；如发现不一致，停止并报告，不继续后续 Task
5. **验证**（如该 Task 涉及代码改动）：
   - 查阅项目 `package.json` 的 `scripts` 与 `AGENTS.md`，确定项目使用的 lint / typecheck / build 命令
   - 运行相关验证命令（如 `pnpm build`、`npm run lint`、`npm run typecheck`），确保改动不破坏现有功能
   - 如 plan 的 Task 中明确给出了验证步骤（如启动服务、访问 URL），按其描述执行
   - 验证失败则停止，报告错误，不继续后续 Task

### 4. 汇总

全部 Task 执行完成后，输出汇总：

- 已执行的 Task 列表与各自结果（成功 / 失败 / 跳过）
- 验证命令的运行情况
- 提示用户：如需保留改动请手动提交 Git（`run` 命令不自动提交）

---

## 格式约束

1. **严格按 plan 执行**：不擅自增加、删减或改写 Task；plan 中未涉及的不做
2. **代码内容原样写入**：plan 代码块中的内容即为要写入文件的最终内容，不要添加注释、调整格式或"优化"
3. **参数来源必核对**：plan 中标注的参数来源是正确性的保证，执行时必须回查来源文件确认
4. **每个 Task 都要验证**：涉及代码改动的 Task 完成后必须运行验证命令；纯文档型 plan 可跳过
5. **失败即停**：任一 Task 执行或验证失败，立即停止，不继续后续 Task，报告问题等待用户决策
6. **不自动提交 Git**：`run` 只执行代码改动与验证，不执行 `git add` / `git commit`，除非用户明确要求

---

## 行为准则

- 执行前完整读取 plan，理解所有 Task 的先后依赖关系
- 并行读取参照文件以提升效率（如 plan 中提到多个对标文件，一次性并行读取）
- 验证命令优先使用项目 `package.json` 中已有的 script
- 执行过程中如发现 plan 描述与实际代码不符（如文件路径错误、函数签名变化），停止并提示用户更新 plan
- 保持与项目现有代码风格一致（缩进、命名、引号等）
